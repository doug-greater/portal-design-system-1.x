Row actions — when to use which

The Menu is the single primitive (a floating list of actions). The split button and kebab are just two triggers that open that same Menu — build the Menu once, then pick the trigger a row needs. (The visible button's intent — primary or error — is just the existing button colors at a slim 28px size.)

Also some unrelated utilities?Add a kebab beside the button — not a second button
No action to push at all?Make the row open a detail / drawer

Every “→ Menu” above opens the same Menu popover — that's why it's one primitive, not one per trigger.

Trigger · Split button

A visible button + a caret that opens the Menu. Use it to encourage one action and offer its related variants in the dropdown. Built at 28px so it sits in a 45px row without growing it. The button's intent — primary or error — is the only thing that changes between the examples below.

Primary steers the happy path. Error is scoped to “the encouraged action fixes a problem” — never a status label (use a Status Badge for that).

Trigger · Kebab (overflow)

A single more_horiz button pinned to the right of the row. Cleanest option, but the lowest discoverability — render it always visible (not hover-only) when its actions matter, and never hide a row's primary action behind it.

In context — and a hard rule on visibility

Row actions are always visible at rest — never revealed on hover. Hover-only actions fail discoverability, touch, and keyboard. Hover may tint the row (and darken a control), but it must never change which controls exist. Every row below shows its actions at rest.

Rules — precise version

1. Default to no control. If a row exists to be opened, make the whole row navigate to its detail / drawer — and do not add a “View”/“Open” button, which only duplicates the row click. A genuine action (Reassign, Export, Duplicate) is different and may still appear. Action controls stop the row click — they perform their action, they don’t also navigate.

2. If a row needs actions, they live in a Menu (the one popover primitive). Decide only how it opens:

• Kebab (more_horiz, pinned right, always visible) when the actions are peers with no single one to encourage, or space is tight.

• Split button (labeled button + caret) when one action is worth encouraging — the button performs it in one click; the caret opens the Menu of its related alternatives.

3. Button intent is the visible button's color: primary for the encouraged happy-path action; error only when that action resolves a problem (clicking it opens the fix). Never use a red button to merely signal a bad state — that is a Status Badge, not a button.

4. One trigger per row, 28px tall, right-aligned in the last column. Unrelated extra actions go inside the Menu, never as a second button. Never hide a row's single most important action behind a kebab alone.

5. Always visible — never hover-only. Row actions render at rest. Do not reveal them on hover (it fails discoverability, touch, and keyboard); hover may tint the row but must not change which controls are present.
